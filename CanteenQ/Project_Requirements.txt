# CanteenQ - Project Requirements Document

> A digital pre-ordering system for school canteens and food stalls

UPDATE: Project status: COMPLETED — Presentation delivered. Cloud deployment is planned as a future update.

---

# 0. MVP SCOPE (CRITICAL DECISION)

### MVP Goal

Deliver a working system where:

* Students can register/login via Firebase (school email validated)
* Students can browse food stalls and menus
* Students can place orders with a 15-minute advance notice requirement
* Students can only place pickup slots within 1 week of the current time and between 7:00 AM and 6:00 PM
* Students can have **only 1 active order per stall** at a time
* Automatic queue numbering by pickup slot and order time
* Staff can view incoming orders in real-time queue
* Staff can update order status (pending → preparing → ready → completed)
* Orders expire and auto-marked UNCLAIMED if not picked up within 10 minutes of pickup slot
* Student can cancel orders up to 15 minutes before pickup
* Complete audit trail for all order transitions

# 1. DOMAIN MODEL (CORE ENTITIES)

## 1.1 User / UserAccount

```java
UserAccount
- id (UUID)
- name (String)
- student_id (VARCHAR, nullable)
- email (VARCHAR, unique)
- firebase_uid (VARCHAR, unique)
- role (ENUM: STUDENT, STAFF)
- created_at (TIMESTAMP)
```

### Rules

* Email validated against school domain (handled at registration layer)
* firebase_uid is the primary identifier from Firebase
* STUDENT role for student accounts
* STAFF role for canteen staff/vendors (admin-created only)

---

## 1.2 Stall

```java
Stall
- id (UUID)
- stall_name (VARCHAR)
- vendor_name (VARCHAR)
- operating_hours (VARCHAR)
- queue_limit (INT, default 100)
- created_at (TIMESTAMP)
```

### Rules

* One stall can have many menu items
* Staff can manage their assigned stall(s)
* Each stall has a default queue limit of 100 orders

---

## 1.3 MenuItem

```java
MenuItem
- id (UUID)
- stall_id (UUID, FK → Stall)
- item_name (VARCHAR)
- description (TEXT)
- price (DECIMAL)
- category (VARCHAR)
- is_available (BOOLEAN)
- created_at (TIMESTAMP)
```

### Rules

* Availability can be toggled by staff in real-time
* Unavailable items cannot be ordered

---

## 1.4 Order / CanteenOrder

```java
CanteenOrder
- id (UUID)
- student_id (UUID, FK → UserAccount)
- stall_id (UUID, FK → Stall)
- total_price (DECIMAL)
- pickup_slot (TIMESTAMP)
- queue_number (INT)
- status (ENUM: PENDING, PREPARING, READY, COMPLETED, CANCELLED, UNCLAIMED)
- created_at (TIMESTAMP)
- updated_at (TIMESTAMP)
```

### Rules

* One active order per student per stall
* Each stall has a default active queue cap of 100 orders
* Student UI shows queue usage and remaining slots
* Queue number generated per stall per day (auto-increment)
* Status transitions: PENDING → PREPARING → READY → COMPLETED/UNCLAIMED/CANCELLED
* UNCLAIMED auto-set by scheduler if not picked up 10 minutes after pickup_slot

---

## 1.5 OrderItem

```java
OrderItem
- id (UUID)
- order_id (UUID, FK → CanteenOrder)
- menu_item_id (UUID, FK → MenuItem)
- quantity (INT)
- subtotal (DECIMAL)
```

### Rules

* Immutable once order is created
* Links the menu item snapshot to the order

---

## 1.6 OrderAuditTrail

```java
OrderAuditTrail
- id (UUID)
- order_id (UUID, FK → CanteenOrder)

- new_status (ENUM)
- changed_by (UUID, FK → UserAccount, nullable for system-initiated)
```

### Rules

# 4. BUSINESS LOGIC (CRITICAL RULES)

## 4.1 Order Placement Rules

* Student must be authenticated (STUDENT role)
* One ACTIVE order per student per stall (active = PENDING, PREPARING, READY)
* Pickup slot must be ≥ 15 minutes in future
* Pickup slot must be within 1 week from now
* Pickup slot must be between 7:00 AM and 6:00 PM
* Each stall has a default active queue cap of 100 orders
* All items must be available
* Menu items must belong to the target stall
* Queue number auto-generated per stall per day

---

## 4.2 Order Cancellation Rules

* Only PENDING or PREPARING orders can be cancelled
* Cancellation window: must be ≥ 15 minutes before pickup_slot
* Audit trail recorded with reason = "STUDENT_CANCELLED"
* After cancellation, student can place new order for same stall

---

## 4.3 Order Status Transitions (Staff)

```
PENDING → PREPARING → READY → COMPLETED
              ↓
           CANCELLED (student cancels)
```

* READY orders auto-transition to UNCLAIMED if not picked up 10 min after pickup_slot
* Only staff can move orders through states
* Each transition audited with changedBy, timestamp

---

## 4.4 Order Expiration Logic (VERY IMPORTANT)

### Spring Scheduler

```java
@Scheduled(fixedDelayString = "${app.orders.unclaimed-check-delay-ms:60000}")
```

Runs every 60 seconds (configurable):

1. Query all READY orders where `now > pickupSlot + 10 minutes`
2. Set each to UNCLAIMED
3. Audit trail recorded with `changedBy = null` (system-initiated)
4. Send SSE notification to student

---

## 4.5 Student Notifications (SSE)

* Student connects to `GET /api/orders/notifications` SSE stream
* Staff updates order status → student receives real-time update
* System auto-marks unclaimed → student receives notification

---

## 4.6 Queue Number Generation

Per stall per day:

```
SELECT MAX(queue_number) 
FROM orders 
WHERE stall_id = ? AND DATE(pickup_slot) = TODAY
```

* Next queue number = MAX + 1 or 1 if no orders yet
* Ensures unique queue numbers per pickup slot

## 5.4 Order Ownership Verification

# 7. EDGE CASES (MUST HANDLE)

## Order Placement

* Student tries to place order for unavailable item → OrderPlacementException
* Student already has active order for stall → OrderConflictException
* Pickup slot < 15 minutes away → OrderPlacementException
* Pickup slot beyond 1 week or outside 7:00 AM to 6:00 PM → OrderPlacementException
* Menu item price is NULL → OrderPlacementException
* Stall doesn't exist → OrderPlacementException

---

## Order Cancellation

* Order already READY → cannot cancel (only within 15 min before pickup)
* Cancellation window expired (< 15 min before pickup) → OrderActionException
* Order not found or not owned by student → OrderNotFoundException

---

## Order Status Updates (Staff)

* Invalid status transition → OrderStatusTransitionException
* Order not found → OrderNotFoundException
* User not STAFF role → InsufficientRoleException

---

## Expiration Scheduler

* Multiple transitions prevented via `@Transactional`
* Only READY orders marked UNCLAIMED
* Audit log has `changedBy = null` for system transitions

---

## SSE Notifications

* Student disconnects and reconnects → reconnect seamless (in-memory hub re-sends)
* Multiple orders → separate events per order

# 13. FINAL ARCHITECTURE SUMMARY

### Simple Student Flow:

```
1. Student logs in via Firebase
2. Student browses stalls/menus
3. Student selects items + pickup slot (≥15 min away)
4. Order placed → queue number assigned
5. Staff sees order in queue, updates status
6. Student gets SSE notification (status change)
7. Student picks up order before 10-min grace period
```

### Simple Staff Flow:

```
1. Staff logs in via Firebase
2. Staff views incoming orders for stall
3. Staff marks order: PENDING → PREPARING → READY
4. Student picks up
5. Staff marks: READY → COMPLETED
   (or auto-marks UNCLAIMED if time expires)
```

### System Automation:

```
Every 60 seconds:
- Check for READY orders past pickup_slot + 10 min
- Auto-mark as UNCLAIMED
- Notify student via SSE
```
